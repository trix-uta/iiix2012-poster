    • Unobtrusive mobile browsing behavior tracking tool

    Matti Lassila, Teemu Pääkkönen, Paavo Arvola, Jaana Kekäläinen, and Marko Junkkari
    University of Tampere, School of information Sciences

    Abstract

    Unobtrusive user tracking is needed in order to achieve realistic test settings for IIR. This demand is stressed, when studying the usage of mobile devices in IR. In this study, we present a model and a tool for observing and recording user’s browsing within a retrieved web document. We extend an existing open source program UsaProxy with functionalities for tracking browsing behaviour.

    1. Introduction

    Mobile devices differ from each other having highly personalized settings and appearance. In addition they are used basically anywhere and any place. While having the same information sources, using a mobile device to access textual information differs from using a desktop device. Namely, in a desktop browser document’s body text fits well to the screen, in most cases horizontally and in many cases vertically. All serious contemporary browsers for the mobile devices aim to remove the cumbersome horizontal scrolling by automatically adjusting the text content for the screen size. Also, vertically, the screen size limits the amount of visible text and thus affects the information access: reaching the relevant material requires browsing within a document, which in the conventional case is basically scrolling the document vertically forth and back. In focused retrieval, to facilitate information access, the user is guided to the most relevant spots provided by a focused retrieval system (see e.g. [2, 3, 7]). Our aim is to model the IR behaviour of users and test the benefits of focused access in a mobile search environment. To study this, a tool for unobtrusive tracking is needed.

    Users interact with websites in several ways, e.g. by clicking hyperlinks, activating input elements such as check boxes or scrolling. Almost all user interaction can be tracked by using a proxy server to inject tracking scripts into the served websites, as demonstrated by Atterer, Wnuk and Schmidt [4]. Their tracking proxy software, UsaProxy, while being useful for tracking user interaction in many cases, requires modifications in order to be useful for tracking what we needed – scrolling and elements visible to the user. In order to track what the user sees at each moment, the appearances and disappearances of website elements - such as images, lists or tables - must be captured (see [1, 9]). To achieve this, we modified the original UsaProxy software to capture scrolling events more precisely while tracking element visibility, and created software tools to extract and visualize this information.

    From the perspective of IIR we introduce tools for observing user’s browsing within a web document. Our tool enables a more realistic test setting, because no disturbing observation is needed and the user is able to perform tasks regardless of the time, place and device. Thus, using the tool we are able to collect empirical data in real and simulated tasks. In addition our tool enables detailed analysis of moves in document browsing, and development of measures based on temporal aspects.

    We specify a formal model for tracking user behaviour while browsing within a result document, i.e. retrieving information from a document. The model gives a general framework for analysing user behaviour in browsing different kinds of documents. In the following section we introduce the model for possible actions in document browsing in a mobile search environment. In Section 3 we introduce the proposed tool, the enhanced UsaProxy, with our contributions on element level tracking and visualization of element browsing. In Section 4 we discuss future directions of our study, such as the use of the suggested tool in IIR testing, and conclude the article.

    2. Model for document browsing

    We define a document browsing session, and its events. These aim to capture the plausible functions of browsing a result document in a mobile search environment. They serve as units in the log analysis. The model assumes certain actions of document browsing, but does not specify user behaviour; rather it facilitates exploring different browsing behaviours.

    A web document’s text body is represented as a sequence of characters. Each character has a position which is calculated from the beginning of the document. A view is the part of the document visible on the screen and it is determined as a part of the document by the start and end positions related to the underlying document. In formal representation a tuple (sequence) is denoted between angle brackets. Square brackets are used to refer to an element of a tuple. For example if t = ⟨a,b,c⟩ is a tuple then t[b] denotes the b element of t.

    Browsing event is a tuple ⟨ps, pe, tsps, tspe, E⟩ where ps and pe are the start and end document positions of the view. In other words, pe - ps is the size of the view. tsps and tspe are timestamps that determine user’s arrival and departure time points to and from the view. In other words, the time the user spends in a view is tspe - tsps. E is a set of visible elements (e.g. images or text elements) in the view. Browsing session is set SB of browsing events.

    Ordered browsing session is a tuple O = ⟨s1, s2,...,sn⟩ where each si (i ∈ {1,...,|SB|}) belongs to SB and sj[tspe] < sj+1[tsps] for each j ∈ {1,..., i-1}. An ordered browsing session is connected if sj[tspe] ~< sj+1[tsps]. The notation "~<" means "only little smaller" or follows immediately (in used precision). O is linear if sj[ps] < sj+1[ps].

    Moving distance M means a sequence between two events in an ordered browsing session. Let sx and sy be two browsing events in O then M(sx, sy) = sy[ps]-sx[ps]. M is a positive integer in forward browsing and negative in backward browsing. Moving is total if the related views do not overlap, i.e. {z| z ∈ sy[ps]} ∩ {z| z ∈ sz[ps]} = ∅. Otherwise it is partial. Jumping means changing a view to another such that they do not overlap. Let si and si+1 be two sequential events in O. If si[pe] +1 < si+1[ps] then the user has jumped forward. If si[ps] -1 > si+1[pe] then the user has jumped backward.

    Scrolling consists of brief adjacent events, where the moving distance is minimal, i.e. the user scrolls with a speed not suitable for reading. Scrolling consists of sequential events denoted by OS =⟨t1, ...,tm⟩, where t1 is called scroll start event and tm scroll stop event. OS is a sequence of events in O started from sk if for each i ∈ {1,...,m} holds ti = sk+i-1 and for each j ∈ {1,..., m-1}: tj[tspe] ~< tj+1[tsps] and tj[ps] ~< tj+1[pe] or tj[pe] ~< tj+1[ps]. In forward scrolling tj[ps] < tj+1[pe] and in backward scrolling tj[pe] < tj+1[ps]. The area between t1[ps] and tm[pe] is called a scroll zone.

    The user can spend different times in a view. For distinguishing different behavior of browsing we denote two threshold values: vb means the threshold value that determines minimal time needed for reading the view. If the used time is less than vb then the view is only blinked. vs means a period of no browsing actions, i.e. it is assumed that the user has other activities. In other words, based on these values we model blinking, reading and sleeping modes.

    A blinked area can be found in a connected ordered browsing session, i.e. it can be distinguish as a subtuple OB of O. OB =⟨t1, ...,t⟩ is a blinked part of O started from sk if for each i ∈ {1,...,m} holds ti = sk+i-1 and for each j ∈ {1,..., m-1}: tj[tspe] - tj+1[tsps] < vb and tj[pe] < tj+1[ps] (in forward browsing) or tj[pe] > tj+1[ps] (in backward browsing). The blinked area is between t1[ps], and tm[pe]. In practice a blinked area is formed by scrolling or jumping.

    A subtuple OR of connected ordered browsing tuple O relates to the read area. OR = ⟨t1, ...,tm⟩ is a read part of O started from sk if for each i ∈ {1,...,m} holds ti = sk+i-1 and for each j ∈ {1,..., m- 1}: vb < tj[tspe] - tj+1[tsps] and tj[pe] < tj+1[ps] (in forward browsing) or tj[pe] > tj+1[ps] (in backward browsing). The read area is between t1[ps], and tm[pe]. It is worth noting that sleeping parts are included in the read parts.

    A browsing event s is in a sleeping mode if tj[tspe] - tj+1[tsps] > vs.

    3. Tracking and visualizing user interaction

    Our tool comprises the browsing model described above, basing itself on a tool called UsaProxy [4]. UsaProxy enables obtaining highly detailed information about the actual browsing of websites, without any interruption or any hint of usage of such tool for the user. The information provided includes the mouse pointer location, entering text, or scrolling a page.

    In its original form, UsaProxy tracks views by having a timed routine check whether the viewed document position has changed at regular intervals. Only a relative vertical position is recorded. This was inadequate for our needs, since we needed to record the positions of both the top and bottom edge of the browser window as well as what elements the user sees at any given time. The open source nature of UsaProxy allowed us to modify its functionality; therefore we decided to devise our own method of tracking and recording all the required information and incorporate the functionality into UsaProxy. Furthermore, there were no tools provided for examining and visualizing the recorded information, so we had to develop our own

    Tracking arrival and departure time points is achieved by capturing scrolling events (i.e. scroll stop and scroll start). Each time the user starts scrolling the website, a scroll start event is triggered. Similarly, when the user stops scrolling, a scroll stop event is triggered. The scroll stop event will only be triggered after the blinking threshold time vb has passed since the last scrolling. New scroll start events will not be triggered before a scroll stop event has occurred. Each scroll stop event demarcates an arrival into a view, while a scroll start event demarcates a departure from a view. A browsing event tuple is constructed by recording the times of these events and the position of the browser window at each event.

    To track the elements that are visible in each view, their positions in the document are stored. When a scroll stop event occurs, current and previous browsing events are compared to figure out a scroll zone that is then compared against the stored element positions. If an element is detected passing the scroll zone and still being in view when the user stops scrolling, an appearance event is recorded for that element. Disappearances are detected by storing the positions of all previously visible elements, and comparing that list to the scroll zone. Similarly to appearances, when a disappearance is detected, a disappearance event is recorded.

    The software writes all tracking information into a log file. Each log line represents a single event that occurred via user interaction on the tracked website (Figure 1). Using our tools, we extract scroll events, element appearances and element disappearances from the log file to create visualization such as the plots illustrated in Figures 2 and 3. In them, the horizontal axis represents the time and the vertical axis the (relative) length of the document. The varying sized curve represents a browsing session, i.e. the view in relation to the document’s vertical length and time. Dark bars represent elements included in the underlying view.

    Blinking, reading and sleeping modes can be seen by examining the browser session plot. For example, in Figure 2 a linear browsing order is presented. At approximately 9:57:19, the page is loaded into browser. At approximately 9:57:28, the user starts scrolling for the first time. From there, the page is scrolled linearly from top to bottom. Each flat section represents a long browsing event, during which the user was reading the document (i.e. reading mode). Diagonal movements represent occurrences where the user was scrolling the document (i.e. blinking mode). At approximately 9:58:03 the user reaches the document's end and stops reading it, therefore entering sleeping mode. In Figure 3 a non-linear browsing session is depicted. There the user returns to the elements seen earlier in the session.

    To allow further refining of the visualized data, the plot is made interactive with zooming, panning and element filtering controls (Figure 4). In Figure 5, for further examination of the logged elements, each element currently visible in the plot can be opened into a dialog window that shows detailed information on the element. In line with Figure 3, the user passes element H5 (having content “An example heading”) very quickly, but returns shortly to see it again. In the illustrated case, the software is set up to track only the appearances and disappearances of heading and image elements.

    The plot style visualization enables one to see when and how the user found certain elements on the website. Interesting elements can be predetermined in the research setting, and the software set up to only track those elements. In case there are several elements or element types of interest, the visualization's element filtering feature can be used to reduce the set of visible elements to any subset of the tracked elements.

    4. Discussion and conclusions

    Interactive information retrieval (IIR) research focuses on users’ interactions with IR systems and information, their search behaviour as well as system evaluations [8]. Empirical studies in this area face several problems, many of which are related to controllability and realism of studies.

    We have introduced a tool for unobtrusive IIR tracking, especially for testing mobile IIR and browsing within a document. Our tool offers

    1. a realistic setting for observing users’ browsing in web documents
    2. a tool for gathering empirical data in real and simulated tasks
    3. a tool for visualizing browsing sessions
    4. means to develop measures based on temporal aspects in browsing documents.

    The tool allows tracking of users’ behaviour with minimal intrusion by the researcher. This enables enhanced realism in test situations but also required control through the detailed logs. Subjects’ action still needs to be triggered, and this calls for subtle consideration of the simulated work tasks.

    The suggested browsing session model and tool are compatible with Borlund’s IIR model [5, 6]. These components pertain to the part 1 of the IIR model ("[a] set of components which aims at ensuring a functional, valid, and realistic setting for the evaluation of IIR systems" [6]).

    In an ongoing study we utilize the tool for exploring how users browse and access information within documents in mobile devices. We investigate browsing styles, which may be connected to work / search tasks. The logs and visualizations provided by the tool will be analysed for browsing patterns.

    The tool can be used for researching different browsing models, like in our study, or exploring user behaviour in different test settings. The tool serves searching for predefined document parts, (known item search) but also allows more open tasks with user defined relevance. One possible application still is to observe and measure the quality of information design of a web page, that is, how the page is constructed e.g. in relation what query keys are used in a web search engine before accessing the page.
